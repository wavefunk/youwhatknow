Getting started

Quickstart

Two things to do. One config file to paste, one command to run. That's genuinely it.

1. Tell Claude Code about youwhatknow

Add the hook config to your user-level Claude Code settings. This applies to all projects, all sessions, all subagents.

~/.claude/settings.json
{ "hooks": { "PreToolUse": [{ "matcher": "Read", "hooks": [{ "type": "http", "url": "http://localhost:7849/hook/pre-read" }] }], "SessionStart": [{ "matcher": "startup", "hooks": [{ "type": "http", "url": "http://localhost:7849/hook/session-start" }] }] } }

2. Start the daemon

terminal
$ youwhatknow

That's it. The daemon runs on localhost:7849, indexes your projects lazily when Claude first touches them, and shuts itself down after 30 minutes of inactivity.

If the daemon isn't running, Claude Code works completely normally. HTTP hooks fail silently — no errors, no disruption. youwhatknow is strictly additive.
Configuration

Daemon Configuration

System-wide settings that apply to the daemon itself. These live in your user config directory and affect all projects.

~/.config/youwhatknow/config.toml optional
# All fields are optional. These are the defaults. port = 7849 session_timeout_minutes = 60 idle_shutdown_minutes = 30
Key Default Description
port 7849 TCP port the daemon listens on. Must match the URL in your Claude Code hooks config.
session_timeout_minutes 60 Minutes of inactivity before a Claude session's read tracking data is cleaned up. Keeps memory tidy during long-running daemon sessions.
idle_shutdown_minutes 30 Minutes with zero incoming requests before the daemon shuts itself down gracefully. Set to 0 to disable (daemon runs forever until killed). The shell hook restarts it automatically on next use.
You probably don't need this file. The defaults are sensible. Only create it if you need to change the port (e.g., conflict with another service) or tweak the shutdown timing.
Per-project

Project Configuration

Per-project settings that control how youwhatknow indexes a specific codebase. These live inside each project and are picked up automatically when the daemon first encounters that project.

your-project/.claude/youwhatknow.toml optional
# All fields are optional. These are the defaults. summary_path = ".claude/summaries" max_file_size_kb = 100 ignored_patterns = []
Key Default Description
summary_path ".claude/summaries" Directory (relative to project root) where TOML summary files and the .last-run commit hash are stored.
max_file_size_kb 100 Files larger than this (in KB) are skipped during indexing. Prevents huge generated files from bloating the index.
ignored_patterns [] Additional glob patterns for files to skip. These are added on top of the built-in defaults (see below).

Built-in ignore patterns

These files are always skipped during indexing, regardless of your config:

always ignored
# Lock files package-lock.json, yarn.lock, pnpm-lock.yaml Cargo.lock, composer.lock, Gemfile.lock, poetry.lock # Generated / minified *.min.js, *.min.css, *.generated.*, *.bundle.*, *.map # Binary files (detected by null bytes in first 512 bytes) # Files exceeding max_file_size_kb
Integration

Claude Code Hooks

youwhatknow uses Claude Code's HTTP hook system. Two hooks do all the work:

PreToolUse / Read

Fires before every Read tool call. The daemon receives the file path and session ID, looks up the file in the project index, and injects a summary into Claude's context. On repeat reads, it adds a warning.

SessionStart / startup

Fires when a new Claude Code session starts. The daemon injects a full project structure map so Claude knows where everything is from the start. No more exploratory file-reading spirals.

What Claude sees

On a first read, the injected context looks like:

injected into Claude's context
-- youwhatknow: src/config.rs -- Configuration loading and validation Public: Config, ProjectConfig, load(), config_dir() Folder: src/ — Core application logic

On a repeat read (2nd, 3rd, ...nth time):

injected into Claude's context
-- youwhatknow: src/config.rs (read 3x this session) -- This file was already read. Summary below — do you still need the full file? Configuration loading and validation Public: Config, ProjectConfig, load(), config_dir()

On session start:

injected into Claude's context
-- youwhatknow: project map -- src/ — Core application logic main.rs — Entry point, daemon startup config.rs — Configuration loading server.rs — Axum HTTP server hooks.rs — Hook handler logic ...
The read always proceeds — youwhatknow never blocks Claude from reading a file. It just injects context so Claude can make a better decision about whether it needs the full content.
API

HTTP Endpoints

All endpoints listen on localhost only. No authentication required (and none supported — it's a local daemon).

POST /hook/pre-read

Called by Claude Code's PreToolUse hook. Receives the hook payload as JSON, returns a hook response with the file summary as additionalContext. The project is identified from the cwd field in the payload.

POST /hook/session-start

Called by Claude Code's SessionStart hook. Returns the project structure map as additionalContext. Triggers lazy project loading if this is the first request for the project.

POST /reindex

Triggers a full re-index of a project. Send a JSON body with a cwd field pointing to the project root. Returns 202 Accepted immediately; indexing runs in the background.

GET /health

Health check. Returns JSON with status and projects (number of loaded project indexes). Useful for checking if the daemon is running.

Under the hood

Indexing & Symbol Extraction

How files are discovered

youwhatknow uses git ls-files to discover files, which means it automatically respects your .gitignore. Files are then filtered by size, binary detection, and ignore patterns.

Symbol extraction

Public symbols are extracted using tree-sitter grammars. This is what populates the "Public:" line in summaries.

Rust

.rs

pub fn, pub struct, pub enum, pub trait, pub type

TypeScript

.ts .tsx

export function, class, interface, type, const

JavaScript

.js .jsx

export function, class, const (ES6 module syntax)

Python

.py

Top-level def, class definitions

Go

.go

Exported (capitalized) func, type, method

Files with unrecognized extensions still get indexed — they just won't have symbol information. The description and folder context still work.

Description generation

File descriptions are generated in two ways:

Primary: Batched calls to claude --model haiku --print. The first 100 lines of each file plus extracted symbols are sent to Haiku in batches of 15. Haiku returns a one-line description per file.

Fallback: If the claude CLI isn't available (or the call fails), descriptions are derived from the filename and symbols. For example: session.rs with symbols SessionTracker, track_read becomes "Session — SessionTracker, track_read".

Incremental re-indexing

On subsequent daemon startups (for a given project), youwhatknow uses git diff against the last-known commit hash to find changed files and only re-indexes those. The commit hash is stored in .claude/summaries/.last-run.

Data

Storage Format

Summaries are stored as human-readable TOML files in each project's .claude/summaries/ directory. You can edit them by hand if you want — the daemon loads them on startup.

Per-folder summary

.claude/summaries/src.toml
generated = "2026-03-21T10:00:00Z" description = "Core application logic" [files."main.rs"] path = "src/main.rs" description = "Entry point, daemon startup" symbols = ["main()"] summarized = "2026-03-21T10:00:00Z"

Project summary

.claude/summaries/project-summary.toml
generated = "2026-03-21T10:00:00Z" last_commit = "abc123" [folders.src] path = "src/" description = "Core application logic"

Nested folders use -- as a separator in filenames: src/indexer/ becomes src--indexer.toml.

All writes are atomic (write to temp file, then rename) so you'll never see a partially-written summary file.

Environment

Environment Variables

All daemon config options can be overridden via environment variables with the YOUWHATKNOW_ prefix:

Variable Equivalent config
YOUWHATKNOW_PORT port
YOUWHATKNOW_SESSION_TIMEOUT_MINUTES session_timeout_minutes
YOUWHATKNOW_IDLE_SHUTDOWN_MINUTES idle_shutdown_minutes

Environment variables take precedence over the config file. Useful for quick overrides without editing TOML:

terminal
# Run on a different port, never auto-shutdown $ YOUWHATKNOW_PORT=9000 YOUWHATKNOW_IDLE_SHUTDOWN_MINUTES=0 youwhatknow
Operations

Lifecycle & PID Management

PID file

The daemon writes its PID to ~/.local/share/youwhatknow/youwhatknow.pid on startup and removes it on shutdown. The shell hook uses this to check if the daemon is already running.

Starting

Just run youwhatknow. If the port is already in use, it exits with an error (another instance is likely running).

Stopping

Three ways the daemon stops:

Ctrl+C / SIGTERM — graceful shutdown. Finishes in-flight requests, cleans up PID file.

Idle timeout — after idle_shutdown_minutes with no requests, the daemon shuts itself down gracefully. Same cleanup as Ctrl+C.

Manual kill — kill $(cat ~/.local/share/youwhatknow/youwhatknow.pid). PID file may be left behind; the shell hook handles stale PID files on next startup.

Multi-project

One daemon handles all projects. When a hook request arrives with a cwd the daemon hasn't seen before, it loads the project config, reads existing summaries from disk, and starts background indexing. No per-project daemon instances needed.

Nix

Nix Integration

The project's flake.nix provides a package and a shell hook. If you use Nix, you can add youwhatknow to your dev shell and have it start automatically.

As a flake input

your-project/flake.nix
# Add youwhatknow as an input inputs.youwhatknow.url = "github:wavefunk/youwhatknow"; # In your devShell packages: packages = [ youwhatknow.packages.${system}.default ]; # Optional: use the shell hook to auto-start shellHook = youwhatknow.devShells.${system}.default.shellHook;

Shell hook behavior

The shell hook checks if the daemon is already running (via PID file). If not, it starts it in the background with disown. It only runs if both claude and youwhatknow are on PATH.

Build caching

The flake uses source filtering — only src/, Cargo.toml, and Cargo.lock affect the build hash. Editing flake.nix, docs, or other files won't trigger a rebuild.
